AMICA: A Private AI Companion for the Elderly, Running Entirely on a $59 Board

Gemma 4 E2B running fully offline on a Qualcomm ARM SBC via llama.cpp — no cloud, no subscription, no data plan, any phone.

Track: llama.cpp Special Technology / Digital Equity & Inclusivity


THE PROBLEM

Loneliness is one of the leading health risks for elderly adults. Studies consistently link social isolation to cognitive decline, depression, and early mortality. At the same time, many elderly people live with early-stage memory loss — forgetting medication schedules, upcoming visits from family, the names of new neighbours. They need a patient, low-pressure way to stay connected and stay organised.

Most AI solutions don't reach this population. Voice assistants require internet. Chatbot apps require app installs, accounts, subscriptions, and reliable data plans. Many elderly users in rural or lower-income settings have none of these. Their carers worry — rightly — about privacy: conversations about health, family, and daily life shouldn't be uploaded to a data centre.

AMICA (AI Memory & Intelligent Care Assistant) is built to serve exactly this group.


WHAT AMICA IS

AMICA is a conversational AI companion that runs 100% offline on a device the size of a deck of cards, requires no app install (any phone browser works), remembers the user's family, medications, and upcoming events across conversations, and costs under $60 to build.

The user connects their phone to AMICA's private Wi-Fi hotspot and opens a browser. That's it. No accounts. No internet. No data ever leaves the room.

A family member or carer can use the companion family portal to pre-load information — the user's name, their medications, who their children are, upcoming appointments. AMICA then references all of this naturally in conversation without being asked.


HARDWARE

The server is an Arduino UNO Q 4GB — a $59 single-board computer built around the Qualcomm Dragonwing QRB2210 (quad-core ARM Cortex-A53 at 2GHz, 4GB LPDDR4, 32GB eMMC, Wi-Fi 5). It runs Debian Linux and hosts everything: the model, the API server, and the Wi-Fi hotspot simultaneously.

The model is Gemma 4 E2B (Q4_K_M quantisation, 3.22GB), served via llama.cpp's llama-server. This is a genuinely resource-constrained deployment — not a large cloud instance pretending to be embedded. The Cortex-A53 has no dotprod, no SVE, no i8mm. Inference runs on basic NEON SIMD and the model fits in RAM with around 780MB to spare.


ARCHITECTURE

The Arduino UNO Q runs llama-server on port 8080 (localhost only), serving Gemma 4 E2B via llama.cpp. On top of that sits amica_server.py, a FastAPI application on port 5000 that injects a personalised system prompt into every request, handles streaming, manages memory persistence, and serves the phone UI and family portal. Both services start automatically on boot via systemd — the device works out of the box with no SSH required.

The phone UI is a single HTML file. Any phone that connects to the AMICA Wi-Fi hotspot can open it in a browser. No app, no install, no account.


HOW GEMMA 4 IS USED

Gemma 4 E2B handles two distinct tasks.

Conversation. Every chat message is sent to Gemma with a compact system prompt (around 180 tokens) containing the user's profile, up to 10 people they know, up to 5 upcoming events, and their current medications. Gemma's instruction-following is strong enough to maintain a warm, patient persona and consistently refer to the user as "you" rather than their name — a small but important detail for elderly users who find being addressed in the third person patronising.

Memory extraction. When a user says something worth remembering ("remind me to meet Karl tomorrow at 6pm", "I met my new neighbour Stacey"), AMICA needs to detect and save it. We use a three-layer cascade.

Layer 1 — structured tag: The system prompt instructs Gemma to end its reply with a structured tag when it detects a save intent. For example: [MEM:person|Stacey|friend|likes gardening] or [MEM:event|2026-05-19|Karl visiting at 6pm]. This is the most reliable path — LLM reasoning consistently outperforms regex on messy, natural-language input.

Layer 2 — regex: A two-pass regex system catches saves that don't produce a valid tag. Pass 1 runs structural patterns on the whole message with no trigger word required, which handles typos like "rememeber". Pass 2 uses explicit trigger detection.

Layer 3 — background LLM: If both layers miss, a compact JSON extraction prompt fires 3 seconds later on the idle llama-server and the result is merged into memory.

One critical Gemma-specific discovery: Gemma 4E defaults to thinking/reasoning mode, spending 100-200 tokens on internal reasoning before producing any visible output. On hardware doing 349ms per token, this adds 60-120 silent seconds before the user sees anything. The fix is the --reasoning off flag on llama-server. This flag must be set explicitly — there is no API parameter that overrides it reliably. Discovering and applying this single flag was the difference between an unusable and a deployable system.


TECHNICAL CHALLENGES

iOS Safari streaming. The phone UI must stream tokens incrementally so the user sees a live response rather than waiting for the full reply. iOS Safari does not support incremental fetch ReadableStream or XHR onprogress — both approaches produced a perpetual loading spinner. The solution is the native EventSource API (GET-based server-sent events). The client POSTs to /api/chat/queue, receives a job_id in under one second, then opens a native EventSource connection to /api/chat/stream/{job_id}. Tokens arrive word by word. SSE keepalive comments are sent every 15 seconds during the silent prompt-processing phase to prevent the connection dropping on mobile networks.

Response latency. With a 180-token system prompt and 349ms per token of processing time, every conversation starts with a 63-second wait before the first token appears. We addressed this with three changes. First, we capped max_tokens at 35 — AMICA's replies average 15-25 tokens, so nothing is cut off, and generation time drops from around 51 seconds to around 14 seconds. Second, we implemented KV cache persistence: after warmup, the slot 0 KV state is saved to disk alongside a SHA256 hash of the system prompt. On the next restart, if the hash matches, the cache is restored instantly — same-day restarts skip the 63-second warmup entirely. Third, any family portal save triggers an immediate background re-warmup, so by the time a carer finishes explaining the portal and hands the phone to the user, the system is already warm.

GPU acceleration. We attempted Vulkan GPU offload via the Adreno 702 GPU and the Mesa Turnip open-source driver. After patching an overly conservative shared-memory check in ggml-vulkan.cpp — which incorrectly rejected Q4_K_M despite its memory requirement being well within hardware limits — the binary loaded successfully. However, the Turnip driver throws a device lost error on every inference request with a real system prompt. The crash is a GPU compute timeout, a known Turnip stability limitation on this specific chip. We fell back to CPU-only, which is stable and predictable.


WHY THESE CHOICES WERE RIGHT

llama.cpp over alternatives. llama.cpp is the only framework that reliably runs a quantised Gemma model on a 4-core ARM Cortex-A53 without requiring a GPU, cloud backend, or specialist hardware. Ollama adds overhead. ONNX Runtime GenAI required 9 separate config files for this model architecture. MediaPipe crashed with GPU assertion failures on device. llama.cpp's server mode provides a clean HTTP API, streaming support, and slot-based KV cache management — all essential for this project.

Browser UI over native app. Any phone, any OS, zero install friction. For an elderly user being handed a device for the first time, this matters enormously. The entire UI is a single HTML file with no framework dependencies and loads instantly over the local hotspot.

memory.json over a database. Plain JSON is human-readable, directly editable by a carer, and requires no migrations, schemas, or dependencies. The family portal is essentially a structured editor for this file.

Offline-first as a feature, not a constraint. AMICA's privacy guarantee — no data ever leaves the room — is what makes it appropriate for elderly care. A cloud-connected alternative would be faster, but it would require consent, data governance, and infrastructure costs that put it out of reach for the target population.


IMPACT

AMICA demonstrates that meaningful AI assistance for vulnerable populations doesn't require expensive hardware or internet connectivity. A $59 ARM board, an open-weight model, and a standard phone browser are sufficient to deliver a private, persistent, genuinely useful companion. The same architecture runs on any Linux SBC — Raspberry Pi 4, Orange Pi, Rock 5 — making it reproducible at scale.

The core thesis: privacy and accessibility are not in tension with capability. You don't need a data centre to give someone's grandmother a companion who remembers her medication schedule and asks how her son Karl's visit went.
